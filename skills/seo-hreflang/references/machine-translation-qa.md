# 机器翻译 QA 标志（2025 年 1 月 QRG）

> 本文件是 `seo-agents` 的中文详细参考，已按当前项目能力边界适配。命令名、API 名、Schema 类型、JSON 字段和专用 SEO 术语按需保留英文。

添加了 Google 2025 年 1 月 23 日的质量评估指南更新
关于**机器翻译内容的明确语言，无人参与
审查**作为大规模内容滥用的一种形式（§4.6.5）。

>“使用自动化工具（生成式人工智能或其他）作为一种省力的方法
> 生成许多几乎没有价值的页面的方法。”

机器翻译的内容很好**经人工审核后
扬声器并更正**。未翻译的 MT — 或“轻微后期编辑”
输出仍然包含幻觉术语、错误的性别/数字
协议，或未翻译的专有名词 - 被视为缩放
内容滥用。

## 表明 seo-hreflang 审核应该浮出水面

|信号|严重性 |笔记|
|---|---|---|
|多个 `hreflang` 替代项指向内容相同（标题 chrome 除外）的 URL |关键|表示正文未翻译；只是模板包装。 |
| `<html>` 上的 `lang="xx"` 属性与肢体语言不匹配 |高|没有最终 QA 步骤的翻译管道输出。 |
|自动翻译的 `<meta name="description">` 长度超过 160 个字符（未修剪）|中等|译者超出了片段限制——没有人工审阅者发现它。 |
| `lang` 属性为 `auto` 或完全缺失 |中等|未声明其语言的页面会使 hreflang + AI 爬虫感到困惑。 |
|正文中散布未翻译的专有名词或产品名称 |低（启发式）|常见MT故障模式；很难自动检测到。 |
| Schema.org `inLanguage` 字段缺失或错误 |中等|多语言审核应交叉检查 `inLanguage` 与正文。 |

## 审计不应该标记什么

- 有一些 MT 页面明确标记为 MT 的网站（a
  人工翻译后备模式）。 Google 的 QRG 明确
  *在诚实标记和明确范围的情况下允许 MT。
- 机器翻译的 UI 字符串 — 这些是 i18n，而不是“内容”。
- 如果审核无法获取回退，则内容为 `lang="auto"`
  信号（保守一点；不要声称我们无法验证的东西）。

## 跨技能委派

- 对于每页 hreflang 验证，请留在 `seo-hreflang` 内。
- 对于更广泛的缩放内容评分（翻译页面的熵，
  体内 AI 模式检测），遵循 `seo-content` via
  `python3 scripts/content_quality.py`。
- 对于“这是由Google自己的自动翻译widget翻译的”
  检测，查找`.goog-te-banner-frame` iframe；自动
  翻译widget明确不受 MT 缩放内容的约束
  滥用但无论如何都会产生较差的段落引用性。

## 主要来源

- 2025 年 1 月 QRG §4.6.5：https://services.google.com/fh/files/misc/hsw-sqrg.pdf
- John Mueller 谈 MT（多集 SOTR，2024-2025）：MT 还可以
  当由人审查时；未经审查的批量 MT 属于滥用行为。

最后验证时间：2026 年 5 月 17 日。

## 本项目适配边界

- 优先使用当前已实现的 `seo-agents` CLI 获取证据；没有 CLI 的能力作为 Agent playbook 或后续扩展处理。
- 不把外部数据源、付费 provider、浏览器渲染或地图 API 写成已执行，除非当前任务确实运行并取得证据。
- 每个 finding 必须有可复核 evidence；数据缺失时明确写“未配置”或“缺少输入”。
