# SERP 重叠方法

> 本文件是 `seo-agents` 的中文详细参考，已按当前项目能力边界适配。命令名、API 名、Schema 类型、JSON 字段和专用 SEO 术语按需保留英文。

## 核心原则

返回相同 Google 结果的两个关键字应由同一页面定位。
返回完全不同结果的两个关键字需要单独的页面。这是
基于 SERP 的聚类的基础——使用 Google 自己的排名决策来确定
内容架构，而不是依赖于关键字文本相似性或词干提取。

## 评分算法

## 第 1 步：收集 SERP 数据

对于候选集中的每个关键字，检索前 10 个有机结果：
- 使用网络搜索或 DataForSEO `serp_organic_live_advanced`
- 仅提取有机结果 URL（忽略广告、特色片段、PAA、知识面板）
- 规范化 URL：剥离协议、尾部斜杠和查询参数（有意义的参数除外）
- 每个关键字存储为一组 10 个 URL

## 第 2 步：成对比较

对于每对关键字（A，B）：
1. 检索 URL 集：`urls_A` 和 `urls_B`
2. 计算重叠：`shared = urls_A intersection urls_B`
3. 得分：`overlap_score = len(shared)`

## 步骤 3：应用阈值

|重叠分数|关系 |行动|
|--------------|-------------|--------|
| 7-10 | **同一篇文章** |将关键字合并到一个目标页面中。使用点击量较高的关键词作为主要关键词。 |
| 4-6 | **相同集群** |放置在同一个spoke簇中。根据数量差异，可能是单独的文章或同一文章。 |
| 2-3 | 2-3 **互连** |放置在相邻的簇中。创建跨集群内部链接。 |
| 0-1 | **分开** |完全不同的集群或从当前pillar主题中排除。 |

## 步骤 4：处理不明确的分数（3-4 范围）

3-4 范围内的分数需要tiebreaker：
1.检查域重叠（相同域但不同页面=更紧密的关系）
2. 检查意图一致性（相同意图类别 = 倾向于同一集群）
3. 检查成交量比率（如果一个关键词的成交量增加了 10 倍以上，那么它可能值得单独发表一篇文章）
4. 当有疑问时，将不同的文章放在同一个集群中（避免凝聚力）

## 优化策略

N 个关键字的完整成对比较需要 N*(N-1)/2 SERP 获取。 40人份
关键词，即780次比较。通过减少不必要的检查来优化：

## 预分组

1. 按意图对所有关键词进行分类（信息性、商业性、交易性）
2. 将共享相同中心词的关键词分组（例如“CRM 软件”变体）
3. 仅在预组内进行成对 SERP 比较
4. 跨组交叉检查边界关键词（每组中的最高量）

## 跳过规则

- 如果关键字 A 和 B 都是同一中心术语的长尾变体并且共享
  相同的意图，假设重叠 4-6（同一集群）而不检查 SERP
- 如果关键字属于不同的意图类别，则假设重叠 0-2，除非它们
  共享一个中心术语
- 通过抽查 SERP 比较来验证假设（对跳过的对的 20% 进行抽样）

## 评分矩阵格式

将重叠数据作为对称矩阵存储在 `cluster-plan.json` 中：

```json
{
  "serp_matrix": {
    "keywords": ["keyword-a", "keyword-b", "keyword-c"],
    "scores": [
      [10, 5, 1],
      [5, 10, 3],
      [1, 3, 10]
    ]
  }
}
```

对角线始终为 10（关键字与其自身完美重叠）。

## 反模式

1. **切勿仅根据文本相似性进行聚类。** “狗训练技巧”和“狗训练
   尽管文本相似，但类”可能具有完全不同的 SERP。
2. **切勿使用仅词干分组。** “Run”和“running”可能针对不同的目标
   完全是意图。
3. **永远不要假设相关搜索属于同一集群。** 使用 SERP 数据进行验证。
4. **永远不要忽略 SERP 功能差异。** 如果关键字 A 触发本地包并且
   关键字 B 触发特色片段，他们可能需要不同的内容类型
   即使有适度的 URL 重叠。
5. **永远不要平等对待所有域。** 维基百科和 Reddit 出现在许多 SERP 中。
   考虑在评分之前过滤掉普遍存在的域（前 5 个最常见的域），或者
   特定领域结果的权重更高。

## 数据源优先级

1. **DataForSEO**（如果有）：最可靠、一致的 SERP 数据。使用
   `serp_organic_live_advanced` 与 `location_code: 2840`（美国）和 `language_code: "en"`。
2. **WebSearch**（后备）：足以进行聚类，但结果可能因会话而异。
   对同一关键字运行多次搜索并使用最常见的结果集。

## 缓存

在单个集群会话中，缓存所有 SERP 结果。如果关键字A的结果
为 A-B 比较获取，将它们重新用于 A-C 比较。这减半
实际需要的 SERP 获取次数。

## 本项目适配边界

- 优先使用当前已实现的 `seo-agents` CLI 获取证据；没有 CLI 的能力作为 Agent playbook 或后续扩展处理。
- 不把外部数据源、付费 provider、浏览器渲染或地图 API 写成已执行，除非当前任务确实运行并取得证据。
- 每个 finding 必须有可复核 evidence；数据缺失时明确写“未配置”或“缺少输入”。
