# 审计综合框架的 10 项原则

> 本文件是 `seo-agents` 的中文详细参考，已按当前项目能力边界适配。命令名、API 名、Schema 类型、JSON 字段和专用 SEO 术语按需保留英文。

这是 seo-agents 用于收集原始结果的规范方法
转化为战略上一致的建议。每一次全现场审核和
深度页面分析在生成之前会先介绍这十个原则
最终行动计划。

这些原则分为四个阶段：

|相|原则|
|---|---|
| **感知** |观察（外部）·观察（内部）·倾听|
| **分析** | THINK · CONNECT (lateral) · CONNECT (system) |
| **验证** |感受·接受|
| **行动** |创造·成长|

未通过所有四个阶段的建议是一个发现，
不是推荐。

---

## 感知

## 1. 观察 — 外部输入

收集信号而不解释它们。对于网站审核来说，这意味着：

- 原始 HTML + 渲染 HTML（通过 `scripts/render_page.py`）
- Schema.org 标记实际存在（通过 `seo-schema`）
- 网站已发布主题的 SERP 可见性（通过 `seo-dataforseo` /
  Google API（如果可用）
- 外链 + 品牌提及景观（来自 `seo-backlinks`）
- 来自 CrUX 的核心 Web Vitals 真实用户数据（通过 `scripts/pagespeed_check.py`）
- 人工智能搜索引用模式（通过 `seo-geo`）
- 目标主要关键字的竞争对手页面

**纪律：** 尚未得分。还没有分类。就收集吧。

## 2.观察——内部元认知

在组装之前审核您自己对场地的假设
建议。 SEO中常见的假设陷阱：

- 假设主页代表网站（通常不代表网站）
  程序化页面或类别页面驱动流量）
- 假设“低流量”意味着“低价值”（意图匹配的低流量可以
  输出转换大量信息查询）
- 假设品牌想要分析师认为的“最佳实践”
  （他们的限制可能是品牌声音、法律或你不知道的权衡
  见）
- 假设 CMS 限制无法修复（通常不是）
- 假设 1.x 的发现仍然适用于 2.x（Google 更新更改了
  地）

**纪律：** 对于每项主要建议，询问“假设是什么
这还靠什么？”如果答案令你惊讶，请将假设提出来
报告，以便用户可以明确拒绝它。

## 3. 倾听——主动接受

了解网站、用户意图和平台信号的实际含义
——不是你期望他们说的。

- 在建议重写之前，请先阅读页面的现有副本。品牌
  语音就是数据。
- 在决定页面类型之前，先阅读 SERP 中的目标关键词
  构建。 SERP 是 Google 对此意图的显露偏好。
- 阅读用户评论/社区讨论/Reddit 主题了解什么
  客户实际询问的问题（与营销团队的想法相反）
  他们询问）。
- 阅读用户之前的对话+记忆（如果有的话）——他们可能会
  已经排除了一些方法。

**纪律：** 如果推荐与 SERP 相矛盾
意图，SERP 获胜，除非你能解释为什么这个网站是
例外。

## 分析

## 4.思考——关键处理

将调查结果简化为首要原则：

- **页面类型**是什么（信息性、事务性、导航性、
  当地、商业调查），目前的布局是否适用
  那个意图？
- AI 功能的**资格下限**是多少（索引 + 可以是
  用片段显示）？如果页面未编入索引，则 AI 不起作用
  还很重要。
- 绑定网站权利的**最高杠杆约束**是什么
  现在？ （通常：单一技术缺陷 — 不可转位、LCP 速度慢、
  缺少规范——这限制了其他一切。）
- **Google 的主要来源指南**对
  推荐？当社区声明与 Google 相矛盾时，请推迟
  至 Google（请参阅 `skills/seo-geo/references/google-ai-optimization-guide.md`）。

**纪律：** 最高杠杆约束首先采取行动
计划，即使它没有“增长”建议那么有趣。

## 5. CONNECT — 横向/关联

结合用户不会自然配对的子技能的发现。
经常产生最高价值推荐的示例：

- `seo-content` 稀薄内容查找 × `seo-cluster` SERP 重叠数据 →
  将三个薄弱页面合并到一个集群中心。
- `seo-schema` 缺少产品架构 × `seo-ecommerce` UCP 未声明
  → 两者都缩小了相同的代理时代购买差距；捆绑为一
  推荐。
- `seo-geo` 低 AI 引用率 × `seo-backlinks` 品牌提及
  权重不足→提及的问题比 AI 的外链多 3 倍
  引文；将链接建设预算重新纳入 PR / Reddit / YouTube 中。
- `seo-technical` SPA 检测 × `seo-content` 缺少主要内容
  → JS 屏蔽的内容是内容查找的上游原因。

**纪律：**任何在连接后仍然存在的子技能发现
没有变化应该持怀疑态度——这可能是一种症状，而不是原因。

## 6. CONNECT — 系统编排

将经过验证的建议连接到可执行序列中：

- 哪个建议**解锁**其他最多？首先这样做。
- 哪些建议相互**依赖**？对它们进行排序。
- 哪些建议可以**并行**？将其表面化为
  用户以便他们可以调度它们。
- 哪些建议需要**尚未安装的工具**（例如
  Firecrawl 用于网站抓取，DataForSEO 用于 SERP 数据）？标记差距。

**纪律：** 行动计划是一个依赖图，而不是一个列表。如果
两个建议不能按任何顺序完成，请这么说。

## 证实

## 7. FEEL——情商+直觉

纯逻辑的建议在与实际读者接触时就中断了/
企业/利益相关者。压力测试针对：

- **用户体验。** 推荐是否会使页面变得更糟
  人类读者？ （常见失败：填充 Google 网站的常见问题解答架构
  甚至没有显示丰富的结果。）
- **品牌声音。** 推荐是否与网站的冲突
  现有的语气？ （常见失败：推荐“答案优先”重写
  在一个使用悬念作为用户体验设备的奢侈品牌上。）
- **操作员能力。** 这对于必须发货的团队来说是否现实？
  它？ （常见失败：向 2 人推荐 30 个新位置页面
  机构。）
- **来之不易的直觉。** 当数据不明确时，信任模式
  来自同一垂直领域过去网站的认可。

**纪律：**如果您无法清楚地表达一个项目的人力成本
推荐，您还没有完全验证它。

## 8.接受——智力上的谦逊

每项建议都应具有可证伪性
诚实：

- 如果建议背后的假设是错误的，会怎样？
  证明一下？ （设置可衡量的检查。）
- 如果用户尝试过此操作但之前不起作用，请显示该操作。
  不要重复推荐同样的东西。
- 如果无法消除限制（法律、品牌、技术），
  推荐必须转向——而不是双倍下注。
- 如果由于 Google 的指导方针发生了变化，v1 推荐现在已经过时，
  明确撤回它。

**纪律：** 每项建议都会得到一个“我们如何知道这一点”
失败了？”线。没有无形的赌注。

## 行为

## 9. CREATE — 生成输出

别再制定战略了。制作神器：

- 一份降价报告，其中包含优先操作、依赖关系和
  可衡量的成果。
- 生成的架构 JSON-LD 准备粘贴到站点中。
- 包含目标关键词、大纲和内部链接的内容简介。
- 当用户要求时，通过 `scripts/google_report.py` 提供 PDF。
- 最高杠杆建议的最小实施，
  不是完整的计划。

**纪律：** 运送工件。分析瘫痪是敌人。

## 10. GROW — 迭代循环

审计只是一个快照，而不是结论。建立反馈循环：

- 通过 `/seo drift baseline <url>` 捕获基线，以便后续
  审计可以证明发生了什么变化。
- 定义用户应监控的一两个领先指标（CrUX
  趋势、目标集群的 GSC 印象、品牌提及增长
  Reddit / YouTube）。
- 安排适合站点速度的重新审核节奏
  （对于高周转电子商务，每周一次；对于 B2B SaaS，每季度一次）。
- 表面化 seo-agents 本身 **无法测量**（离线
  转化、品牌提升、客户访谈），以便人性化关闭
  那些循环。

**纪律：** 每次审核的最后一段命名接下来的内容
审计应寻找。

## 如何调用框架

每次全站点审核 (`/seo audit`) 和深页审核 (`/seo page`)
在发出之前先经历“感知”→“分析”→“验证”→“行动”
行动计划。关键/高/中/低优先级存储
发生在验证阶段**之后**，而不是代替它。

单一用途命令（`/seo schema`、`/seo images`、`/seo technical`、
等）当用户询问一个狭窄的问题时可以跳过整个循环
- 但他们的建议至少应该通过思考+接受
在被发射之前（这是否基于合理的第一原则，并且是
可证伪性浮出水面？）。

## 何时升级给用户

这些原则是 seo-agents 的；它们不是用户的。将它们浮现出来
对于以下情况的用户：

- 推荐需要接受你不愿意接受的假设
  （CONNECT 横向经常产生这些 - 表面链接并让
  用户确认）。
- 验证阶段标记了品牌声音/运营商能力/硬性
  您可以看到但无法解决的约束。
- 审计发现没有上游限制，并建议
  优化可能为时过早。

## 本项目适配边界

- 优先使用当前已实现的 `seo-agents` CLI 获取证据；没有 CLI 的能力作为 Agent playbook 或后续扩展处理。
- 不把外部数据源、付费 provider、浏览器渲染或地图 API 写成已执行，除非当前任务确实运行并取得证据。
- 每个 finding 必须有可复核 evidence；数据缺失时明确写“未配置”或“缺少输入”。
